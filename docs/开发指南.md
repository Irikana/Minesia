## 名称来源

Minesia（又名创世），名字来源是取 Minecraft 的前 5 个字母与 Principia 的后 3 个字母构成，但考虑发音，后来变成了 Minecia，又因为可能引起歧义，把 c 换成了 s，最后变成 Minesia。

## 开发约定

开发 Minesia 时，应当注意一些约定俗成的规则，以免造成代码污染，以及管理混乱。

### 1. 新物品注册规则

当添加新物品时：

- **伤害范围**：如果物品具有伤害，应当是一个范围，请将该特性注册到 `scripts/random_damage` 下
- **体力消耗**：如果物品具有体力消耗，请将该特性注册到 `scripts/stamina` 下
- **自定义事件**：如果有自定义事件，记得注册到 `scripts/custom_events` 下
- **暴击率**：如果物品有暴击率加成，请将该特性注册到 `scripts/critical_hit` 下
- **Lore 描述**：所有物品属性描述都应当通过语言文件配置（详见第13条 Lore 描述规范）

### 2. 功能模块管理

| 功能类型 | 管理目录 | 说明 |
|----------|----------|------|
| 随机伤害（伤害） | `scripts/random_damage` | 所有伤害相关逻辑 |
| 体力消耗 | `scripts/stamina` | 所有体力相关逻辑 |
| 套装效果 | `scripts/set_effect` | 包括一切对槽位具有要求的自定义事件的触发事件 |
| Lore 描述 | `RP/texts/*.lang` | 所有物品的描述通过语言文件静态配置 |
| 调试系统 | `scripts/debug` | 调试模式管理，支持动态开启/关闭 |
| 暴击系统 | `scripts/critical_hit` | 暴击率管理与暴击伤害计算 |

### 3. 价值映射关系

附加包里有默认约定好的价值映射关系：

#### 货币价值

| 货币 | 等价关系 |
|------|----------|
| 1 个绿宝石币 | ≈ 8 个钻石币 |
| 1 个钻石币 | ≈ 9 个金币 |
| 1 个金币 | ≈ 8 个银币 |
| 1 个银币 | ≈ 11 个石币 |
| 1 个石币 | ≈ 14 个木币 |

> ⚠️ **注意**：红宝石币作为神秘的非法货币，不具备用其他货币衡量价值的能力，且万万不可在未特意说明时，为它创造任何可能的获取途径。

#### 物品价值

| 物品 | 等价关系 |
|------|----------|
| 1 个腐肉 | ≈ 1 个木币 |
| 1 个铁锭 | ≈ 4 个石币 |
| 1 个金锭 | ≈ 5 个银币 |
| 1 个钻石 | ≈ 3 个金币 |
| 1 个下界合金锭 | ≈ 4 个钻石币 |

> 📝 **说明**：价值映射关系仅仅代表着一种配置参考，而非固定的关系，附加包中的内容在合理范围内允许价值波动以增加可玩性。

### 4. 目录说明

| 目录 | 说明 |
|------|------|
| `builds/` | 仅用于 bridge 快速构建，本质上不会影响任何开发和调试进程，在开发创世时应当忽略 |

### 5. 子包同步规范

附加包向玩家提供难度选择，目前通过子包来实现。每当开发或修改结构性内容时，应当注意保持子包一起更新，在没有强调的情况下，子包与主包更新内容应同步。

**当前子包结构：**

| 子包名称 | 路径 | 说明 |
|----------|------|------|
| Easy | `BP/subpacks/Easy/` | 简单难度（奖励增加25%） |
| Very Easy | `BP/subpacks/Very Easy/` | 非常简单难度（奖励增加50%） |

> ⚠️ **注意**：修改主包中的脚本逻辑时，务必检查子包中是否存在相同文件，如存在则需同步更新。

### 6. 武器类型特性

新增武器时，如果声明了镰刀特性（在物品名称或描述或自定义事件中），在加入到附加包时应当按具体情况选择自定义事件中的镰刀横扫事件。大多数时候加入一把镰刀武器，都默认其能够触发镰刀横扫事件，此为武器类型特性，如果没有特意声明不需要添加至 lore。

#### 材料对应武器伤害关系

不同材料对应不同武器类型时，伤害范围应遵循以下规律：

| 武器类型 | 伤害计算公式 | 说明 |
|----------|-------------|------|
| 剑 | x（固定值） | 原版剑伤害作为基准 |
| 镰刀 | x+1 ~ x+x/2 | 即 x+1 到 1.5x |

**示例对照表：**

| 材料 | 原版剑伤害(x) | 镰刀伤害范围 |
|------|--------------|-------------|
| 木/金 | 4 | 5~6 |
| 石 | 5 | 6~7.5 |
| 铁 | 6 | 7~9 |
| 钻石 | 7 | 8~10.5 |
| 下界合金 | 8 | 9~12 |

> 📝 **说明**：实际配置时可在合理范围内微调，但应保持镰刀伤害高于同材料剑的伤害这一基本原则。

### 7. 调试规范

由于很多控制台输出代码不工作，请在添加调试内容时遵循以下规则：

- **控制台输出**：将控制台输出换成发送聊天信息（使用 `world.sendMessage()` 或 `player.sendMessage()`）
- **自定义指令调试**：如果需要调试或使用自定义指令，请使用 `scriptevent` 数据驱动事件

#### 调试管理系统

项目内置了调试管理系统（`scripts/debug`），支持动态开启/关闭调试模式，方便开发调试与发布切换。

##### 调试模式开关

| 指令 | 说明 |
|------|------|
| `scriptevent minesia:debug_on` | 开启调试模式 |
| `scriptevent minesia:debug_off` | 关闭调试模式 |
| `scriptevent minesia:debug_status` | 查看当前调试状态 |

##### 使用方法

```javascript
import { debug } from "./debug/debugManager.js";

// 调试日志（仅在调试模式下输出）
debug.log("这是一条调试信息");
debug.logPlayer(player, "向玩家发送的调试信息");

// 带模块标签的调试日志
debug.logWithTag("随机伤害系统", "伤害计算完成");
```

##### 开发流程建议

1. **开发阶段**：使用 `scriptevent minesia:debug_on` 开启调试模式
2. **调试完毕**：使用 `scriptevent minesia:debug_off` 关闭调试模式
3. **发布前**：确保调试模式已关闭，避免影响玩家体验

### 8. 文档术语规范

当撰写项目相关 md 文件（如更新日志、技术文档等）时，请保证术语和专有名词的保留以及规范，统一使用项目内约定的名称：

- **创世等级**：应使用 "Minesia Level" 而非 "Creation Level"
- **项目名称**：统一使用 "Minesia" 或 "创世"
- 其他专有名词请参照项目内已有的命名规范

> 📝 **说明**：保持术语一致性有助于维护文档的专业性和可读性，避免因翻译或命名不统一造成的混淆。

### 9. 测试模组规范

在测试模组时，遇到问题我会撰写 `changelogs` 下的 `.plan.md` 文件，其中：

- **待执行项标记**：带 `*` 的项为待执行项，执行后应去掉 `*` 号
- **更新提示处理**：如果提示语中有要求按照文件进行更新或改动，应酌情去掉或添加 `*` 号

### 10. 构建发布规范

当提示词中声明需要"构建"时，请按以下流程执行：

#### 构建前检查

1. 检查 `changelogs/zh_CN/` 目录下最新的 `plan.md` 文件
2. 确认所有待执行项（带 `*` 号）已全部完成并去掉 `*` 号

#### 构建流程

1. **确定日期**：将中文 `plan.md` 中的未定日期改为执行当天的北京时间
2. **创建英文更新日志**：在 `changelogs/en_US/` 目录下新建对应版本的英文版更新日志
3. **重命名计划文件**：将 `plan.md` 重命名为 `.md`（去掉 plan 前缀）
4. **更新清单版本号**：将版本号更新到 BP 和 RP 的清单文件（`manifest.json`）中
5. **构建附加包**：将 BP 和 RP 构建为 `Minesia [版本号].mcaddon` 文件
6. **创建下一版本计划**：在 `changelogs/zh_CN/` 目录下新建下一个版本号的 `plan.md`

### 11. 版本号规则

版本号格式为：`长期版本号.主要版本号.寻常版本号`（如 `0.0.13`）

#### 版本号说明

| 版本号类型 | 说明 | 更新频率 |
|------------|------|----------|
| 长期版本号 | 在相当长一段时间内保持不变 | 极少更新 |
| 主要版本号 | 仅在较大更新或重大更新时增加 | 不频繁 |
| 寻常版本号 | 任何更新都可以增加 | 最频繁 |

#### 版本号递增规则

- **寻常版本号**：每次更新默认增加此版本号
- **主要版本号**：增加时，寻常版本号清零重新开始
- **长期版本号**：增加时，主要版本号和寻常版本号同时清零

> 📝 **说明**：当提示词没有特意声明时，默认只增加寻常版本号。

### 12. 更新日志记录规范

每次开发或修改内容时，应当将改动记录到 `changelogs/zh_CN/` 目录下对应版本的 `plan.md` 文件中：

- **记录时机**：在进行任何开发或修改之前，先在计划更新日志中记录待执行项
- **待执行项标记**：带 `*` 的项为待执行项，执行后应去掉 `*` 号
- **格式要求**：每个段落之间需要空一行，确保 Markdown 渲染时能正确换行
- **语言风格**：更新日志应当口语化，用自然的表达方式描述改动，避免过于正式或生硬的措辞

#### 更新日志格式示例

```markdown
# 版本号

## 日期

***

### 分类标题

段落内容...

空一行后写下一段...

### 另一个分类

更多内容...
```

### 13. Lore 描述规范

物品的描述通过语言文件（`RP/texts/*.lang`）静态配置，无需脚本运行时处理，确保零性能开销。

#### 描述格式

使用 `~LINEBREAK~` 换行符实现多行描述，格式如下：

```
item.minesia:物品ID=物品名称~LINEBREAK~~LINEBREAK~描述内容
```

> 📝 **说明**：`~LINEBREAK~~LINEBREAK~` 在物品名称后创建一个空行，贴合原版风格。

#### 颜色代码

| 颜色 | 代码 | 用途 |
|------|------|------|
| 蓝色 | `§9` | 主要属性（攻击伤害、套装名称、副手装备提示） |
| 红色 | `§c` | 体力消耗 |
| 黄色 | `§e` | 暴击率 |
| 灰色 | `§7` | 次要描述（效果详情、趣味文本） |

#### 描述顺序

物品描述应按以下顺序排列：

1. **攻击伤害**（§9）- 武器专属
2. **体力消耗**（§c）- 武器专属
3. **暴击率**（§e）- 有暴击加成的物品
4. **套装/副手效果**（§9 标题 + §7 详情）
5. **自定义事件效果**（§7）
6. **趣味描述**（§7）

#### 示例

```
item.minesia:black_dagger=黑色匕首~LINEBREAK~~LINEBREAK~§9+2~7 攻击伤害~LINEBREAK~§c+2 体力消耗~LINEBREAK~§e+50% 暴击率~LINEBREAK~§7攻击时25%概率使目标凋谢2秒~LINEBREAK~§7这把匕首看起来很危险。
```

显示效果：

```
黑色匕首

+2~7 攻击伤害 (蓝色)
+2 体力消耗 (红色)
+50% 暴击率 (黄色)
攻击时25%概率使目标凋谢2秒 (灰色)
这把匕首看起来很危险。 (灰色)
```

#### 添加新物品描述

1. 在 `RP/texts/zh_CN.lang` 添加中文描述
2. 在 `RP/texts/en_US.lang` 添加英文描述
3. 无需修改 `scripts/lore_system` 中的任何文件

> ⚠️ **注意**：`scripts/lore_system` 目录已废弃，仅保留框架代码以备未来动态内容需求。所有静态描述均通过语言文件配置。

## 技术限制与疑问处理

在开发过程中，如果遇到以下情况，应当及时停下来向用户询问：

- **技术限制**：发现 Script API 不支持某些功能（如熔炉烧炼事件、特定实体事件等）
- **实现方案不确定**：存在多种可行方案，需要用户选择
- **需求不明确**：对功能需求理解有歧义
- **依赖问题**：需要用户确认是否引入新的依赖或修改现有结构

> 📝 **说明**：遇到问题及时沟通，避免做无用功或产生不符合预期的实现。
